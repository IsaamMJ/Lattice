# Lattice v0.7 — Stable fingerprints + close-reason taxonomy

**Status:** shipped in v0.7.x (implemented). Replaces the abandoned v0.7 decision-tracking layer (commit `e2a848a`). The canonical, current finding schema lives in [`finding-schema.md`](finding-schema.md) — this spec is the design record for the id algorithm + close-reason taxonomy as built.

**One-line summary:** redefine the `id:` algorithm so a finding's identity survives line shifts, flatten the `open/<date>/` and `closed/<sha>/` directories so re-audits dedup at the filesystem level, and require a closure reason on `lattice-close.sh`.

## Motivation

External research against Semgrep, GitHub CodeQL, DefectDojo, Snyk, and SARIF surfaced one consensus best practice and one Lattice-specific gap:

1. **Consensus**: stable finding identity uses **multi-part fingerprints** (rule + location-context hash + surrounding code), explicitly *not* line numbers. Lattice's current `id = sha1(rule + module + file + line)[:12]` is fragile — a single line shift produces a "new" finding indistinguishable from a real new defect.
2. **Lattice-specific**: the `open/<sweep-date>/` directory structure means re-running an audit on a new date creates a fresh file in a new directory, even though the slug (`<TIER>-<module>-<rule>.yml`) is stable. Filesystem dedup is broken by the date partition. Same problem with `closed/<sha>/` for closures.

Result without v0.7: rerunning `/security-audit` on jiive-Lumi a week later doubles the open-findings count. The pilot will hit this on its first re-audit.

## What stays (don't dilute)

- **`tests:` + `simulate:` fields remain Lattice's differentiator.** No mature tool requires repro evidence to close. Keep requiring it where the workflow already does.
- **Manual closure** (vs Semgrep-style auto-close-on-absence). Verify-don't-eyeball is the philosophy.
- **YAML-per-finding, regen-CLAUDE.md, partial-fix lifecycle** (v0.6.3) — unchanged.
- **`tests:`/`simulate:`/flow/coverage** (v0.6.4) — unchanged.

## Change 1 — fingerprint algorithm

Redefine the existing `id:` field. Same name, new inputs.

### New algorithm

```
id = sha1(
  dimension + "\n" +
  rule + "\n" +
  file + "\n" +
  code_context_normalized
)[:12]
```

**Inputs:**
- `dimension` — `audit | scale | security | flow | coverage`
- `rule` — kebab-case rule slug (already required)
- `file` — path relative to project root (already required)
- `code_context_normalized` — see below

**Removed from inputs:** `module` (subsumed by `file`), `line` (the bug).

### Code-context selection

1. Open `file` at the project root.
2. Start at `line:`. Walk forward (and forward only) collecting **the first 2 non-blank, non-comment-only lines** beginning at or after `line`.
3. If fewer than 2 such lines exist (EOF), pad with empty strings to keep hash length stable.
4. Normalize each line: strip leading/trailing whitespace; collapse internal whitespace runs to a single space; preserve case.
5. Join with `\n`.

Comment-only detection is best-effort by file extension: `//`, `#`, `--`, `/*`, `*` at line start. Mixed code+comment lines are kept (the code is the anchor).

### Why this survives refactors

- Single-line shift: same context lines, same hash. ✓
- Reformat (e.g. line-break a long call): the *first* 2 non-blank lines after the original `line:` are likely still the same tokens. ✓ (best-effort; not perfect)
- Variable rename inside the line: hash changes — treated as a new finding. **Acceptable**: a meaningful rename probably warrants re-review.
- File rename: hash changes — new finding. Matches every other tool's behavior.
- Whole-file rewrite: hash changes — new finding. Correct.

### Stored alongside (debugging aid)

Add an optional `code_context:` field to the YAML capturing the exact 2-line slice used. Pure transparency — never re-read at validate time.

```yaml
id: a1b2c3d4e5f6
code_context: |
  router.get('/users/:id', async (req, res) => {
    const user = await getUser(req.params.id);
```

## Change 2 — flatten directory structure

### Before (v0.6.x)

```
.lattice/findings/
├── open/
│   ├── 2026-05-02/
│   │   └── CRITICAL-payments-webhook-timing-unsafe-eq.yml
│   └── 2026-05-09/
│       └── CRITICAL-payments-webhook-timing-unsafe-eq.yml   # ← duplicate of the above
└── closed/
    └── abc1234/
        └── CRITICAL-admin-token-eq.yml
```

### After (v0.7)

```
.lattice/findings/
├── open/
│   └── CRITICAL-payments-webhook-timing-unsafe-eq.yml       # one file per finding, period
└── closed/
    └── CRITICAL-admin-token-eq.yml
```

### Replacement YAML fields

The information previously encoded in directory names moves into fields:

```yaml
first_seen_sweep: <sweep_id>            # was: open/<date>/ encoded the date
first_seen_date: <ISO date>             # human-readable mirror, optional
last_seen_sweep: <sweep_id>             # updated on every re-audit that re-finds this id
last_seen_date: <ISO date>

closed_by_commit: <7-char-sha>          # already exists; was also encoded in closed/<sha>/
closed_at: <ISO timestamp>              # already exists
```

Sweep manifests (`.lattice/findings/sweeps/<sweep_id>.yml`) already track `opened` / `unchanged` / `closed_since_last` / `regressed` per sweep. That's where "what came in this sweep" lives — not in the path.

### Re-audit semantics

`lattice` audit commands now:
1. Compute `id` for each candidate finding.
2. If `.lattice/findings/open/<slug>.yml` already exists with the same `id`, **update `last_seen_sweep` / `last_seen_date` only**. Don't overwrite triage state.
3. If the slug exists with a **different** `id`, that's a slug collision — fail loudly. The slug must be made more specific (rule needs a sub-tag).
4. If the slug doesn't exist, create the file with `first_seen_sweep` / `last_seen_sweep` both pointing at this run.

Closed findings (`.lattice/findings/closed/<slug>.yml`) are checked first: if the new finding's `id` matches a closed file, that's a **regression** — move closed → open with `previously_closed_in:` set (existing reopen flow).

## Change 3 — close-reason taxonomy

`lattice-close.sh` currently accepts free-form prose. Add a required `--reason`:

```
bash scripts/lattice-close.sh <slug> --reason <fixed|false-positive|wont-fix|out-of-scope|duplicate> [--evidence "<text>"]
```

Reasons:
- **`fixed`** — code changed, defect gone. Default. `tests:` / `simulate:` from the YAML become the verification spec, run as part of validate.sh in CI when present.
- **`false-positive`** — finding was wrong; the code is correct. Requires `--evidence` (one line explaining why the auditor was wrong).
- **`wont-fix`** — accepted risk. Requires `--evidence` (justification). Equivalent to existing `status: wont_fix` — moves to closed/.
- **`out-of-scope`** — finding belongs to a different module/team. Requires `--evidence`.
- **`duplicate`** — same as another fingerprint. Requires `--evidence` set to the other finding's `id`.

YAML field added on close:

```yaml
closed_reason: fixed | false-positive | wont-fix | out-of-scope | duplicate
closed_evidence: <text>     # required for any reason except `fixed`
```

CLAUDE.md regen surfaces non-`fixed` closures distinctly:

```markdown
## Recently closed (last 7 days, 12)
- [x] `payments` / `webhook-timing-unsafe-eq` — fixed in abc1234 on 2026-05-06
- [~] `results` / `stack-trace-leak` — wont-fix in def5678 on 2026-05-05 — "low blast radius, sentry already filters"
- [?] `admin` / `phantom-rate-limit` — false-positive in 9876fed on 2026-05-04 — "rate limit lives in nginx, not app"
```

## Migration

One-shot script: `scripts/migrate-v0.7.sh`.

For each YAML in `.lattice/findings/{open,closed}/**`:
1. Recompute `id:` per the v0.7 algorithm. Preserve old id under `legacy_id:` for one minor cycle.
2. Move file from `open/<date>/<slug>.yml` → `open/<slug>.yml`. If a destination slug already exists, the second one wins iff it has the more recent `sweep_date`; the older one is logged + dropped (it was already a duplicate).
3. Move file from `closed/<sha>/<slug>.yml` → `closed/<slug>.yml`. Same dedup rule.
4. Synthesize `first_seen_date` from the old directory name; `first_seen_sweep` from `sweep_id` if present, else `migrated`.
5. Synthesize `closed_reason: fixed` for all existing closed findings (the only reason that existed pre-v0.7 by convention). `closed_evidence:` left empty for migrated `fixed` closures (no requirement applies).
6. Compute `code_context:` and write it for every finding (best-effort; if the file/line no longer resolves cleanly, write `code_context: |` empty and log a warning — finding still validates because `code_context` is optional).

Migration is idempotent: running twice produces the same output and exits 0.

Validate: `scripts/validate.sh --post-migration` runs regen-check + structural-drift gates + sweep-manifest integrity. Same gates as v0.6.4.1, no new ones required.

## Compatibility window

- v0.7 ships with the migration script + a hard deprecation: validate.sh fails if it sees `open/<date>/` or `closed/<sha>/` directories *and* the migration script hasn't been run. Operators must run migration once.
- `legacy_id:` field is read for one minor (v0.7.x) and dropped in v0.8.

## Out of scope for v0.7

- **Auto-close on absence**. Tools like Semgrep auto-close findings that don't appear in the next scan. Lattice deliberately doesn't — requiring a manual close + evidence is the moat.
- **Cross-dimension dedup**. Already on the v0.8 roadmap (per `finding-schema.md`). v0.7 dedups within a dimension; cross-dimension same-defect detection waits.
- **SARIF export**. Same call as v0.6: not yet, ship if/when GitHub code scanning integration is asked for. The new fingerprint algorithm is *closer* to SARIF's `partialFingerprints` though — exporter will be a thin shim when it's needed.
- **Rebuilding decision-tracking**. The dead v0.7 stays dead. If we ever revisit constraint-bearing-doc tracking, point the old hook infrastructure at `docs/*.md` with `affects:` frontmatter — different scope, same code.

## Validation plan (gate for shipping v0.7)

1. **Synthetic test**: take an existing finding, shift its `line:` ±5, confirm `id` stays stable.
2. **Refactor test**: rename a variable in the anchor lines, confirm `id` changes (correct: re-review warranted).
3. **File rename test**: `git mv` the file, confirm `id` changes (correct: new finding).
4. **Re-audit test on jiive-Lumi**: run `/security-audit lumi` twice in succession on the same commit. Expect zero new findings on the second run, all `last_seen_sweep` updated.
5. **Cross-commit re-audit on jiive-Lumi**: run `/security-audit lumi`, make an unrelated change to the file (add a logging line above the finding), re-run. Expect the finding to be re-recognized, not duplicated.
6. **Migration test**: run `migrate-v0.7.sh` on a copy of the current jiive-Lumi `.lattice/`, confirm validate.sh passes post-migration and CLAUDE.md regenerates byte-identical content (modulo the new closed-reason markers).

If any of 1–6 fails, ship v0.7-rc; do not promote until 6/6 pass.

## Sequencing

1. **First**: pilot `/flow-audit` on jiive-Lumi using v0.6.4.1 (still the gate; this spec doesn't unblock that).
2. **If the pilot reruns the audit even once**, v0.7 fingerprinting becomes urgent — ship next.
3. Bundle close-reason taxonomy into the same release. Cheap, complementary.
4. Flow default-on inclusion in `audit-sweep` is a separate decision based on pilot signal — ship as v0.7.1 or v0.8 depending on what the pilot says.
