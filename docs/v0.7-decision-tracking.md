# Lattice v0.7 — Decision Tracking Specification

**Date:** 2026-05-03  
**Status:** Ready for Implementation  
**Audience:** Engineering team (implementation starting v0.7)  
**Based on:** Ultra-minimal design after ChatGPT and Lattice review iterations

---

## Executive Summary

Lattice v0.7 adds **decision tracking** to the Lattice audit framework. This feature keeps architectural decisions (the WHY) linked to code, warns developers when they're about to break a decision, and prevents decision rationale from being lost between sprints.

**One killer moment:** Developer refactors webhook code 6 months later and gets warned: *"Decision 'async-webhooks' references src/webhooks/. Still valid? Review?"*

This is not a generic documentation tool. This is a lightweight guardrail: decisions find developers when they're about to break them, giving them a chance to review and decide.

---

## Problem Statement

### Why Decisions Get Lost

You inherit code. You have no idea why it was designed this way.

- "Why async webhooks?" → Lost in Slack, old PRs, or someone's head
- "Why this rate limit?" → Scattered across docs, maybe outdated
- "Why not use Redis here?" → Forgotten after the sprint ends

**Result:** Developer re-invents the wheel, makes wrong choices, or duplicates work.

### Why Existing Approaches Fail

**Architecture Decision Records (Nygard, 2011)** are the prior art. They work for design teams that maintain discipline. But:
- **Adoption friction:** Developers write ADRs → nobody reads them → they rot
- **No awareness:** ADRs warn at code review time (already expensive) not while coding
- **Fragmentation:** ADRs live in a separate folder, not linked to code
- **Context loss:** When a junior developer refactors, they don't see the decision warning

Lattice v0.7 fixes all three:
1. **Lightweight guardrail:** Warn developers when they're about to break a decision (while coding, not after)
2. **Durable linking:** Link at file/module level, not line numbers (survives refactors)
3. **Automatic detection:** Pre-push check, one-time signal per commit, no spam

### Current State: Lattice v0.6 (Reactive)

Lattice v0.6 *finds* drift after code ships:
- Audits code against docs (TTD, DDD, CLAUDE.md)
- Grounds findings with file:line evidence
- Used on jiive-backend: 69 findings (19 HIGH, 33 MEDIUM, 17 LOW)
- Problem: Fixing drift is expensive (rework + re-audit)

**Gap:** No prevention layer. Decisions decay because there's no warning system.

### New Layer: v0.7 (Proactive)

Lattice v0.7 adds decision tracking:
- Developers write **decision.md** (what we decided + why)
- System links decision to code (file/module level)
- When code changes → system warns (pre-push)
- Developers review: "Still valid? Update description?"

**Result:** Decisions stay current, drift prevented, developers understand intent.

## When to Write a Decision (Adoption Trigger)

This is critical. Without clear trigger rules, developers won't write decisions and the system dies.

**Write a decision when:**

1. **New module created** — Any new top-level folder in `src/`
   - Example: Creating `src/webhooks/` → Write decision about webhook strategy
   
2. **Major architectural choice** — Picking between fundamentally different approaches
   - Example: "Async vs. sync webhooks" → Write decision with tradeoffs
   
3. **Tradeoff decision** — Chose X over Y, and you want to explain why
   - Example: "Redis streams vs. in-memory queue" → Write decision explaining scale assumption

4. **Constraint or assumption** — Code has a hidden limit that could break in the future
   - Example: "Max 10k pending webhooks in memory" → Write decision explaining why

5. **Non-obvious design** — Code will confuse someone in 6 months
   - Example: "Why are we encoding JSON twice?" → Write decision explaining the reason

**Don't write a decision for:**
- Bug fixes (unless the fix reveals a flawed assumption)
- Refactors (unless you're changing the approach)
- Feature additions (unless they introduce architectural trade-offs)

**Guideline (rough heuristic):** Start with 1-2 per sprint. Adjust based on what actually gets re-surfaced by warnings. If you're writing zero decisions, the system failed.

## Solution Design

### Three Core Components

#### 1. Decision Format (decision.md)

Simple markdown + YAML metadata. Written once per decision.

```yaml
---
title: "Async Webhooks"
why: "Sync model hits 500ms timeout at 10k/day. Async buys us headroom to 100k/day."
status: active
created: 2026-01-15
updated: 2026-05-03
affects:
  - src/webhooks/
  - src/queue/
  - tests/webhooks.test.ts
alternatives: |
  Considered: Redis streams (requires new dependency, operational overhead)
  Considered: Database queue (slower, no ordering guarantees)
  Chose: Async/await + in-memory queue because simplicity wins for current scale
---

## Why

At 10k/day webhook volume, sync processing hits our 500ms timeout.
Async + in-memory queue lets us:
1. Return immediately (reduces customer timeout errors)
2. Process at queue pace (can backpressure without blocking API)
3. Scale to 100k/day before needing external queue system

## Implementation

- Use Node async/await
- In-memory queue with drain on shutdown
- Log failures to DynamoDB for replay
- No external dependency (Redis/Kafka) yet

## Constraints

- This works for stateless webhooks only
- If webhook order matters → must rethink
- Memory limit: ~10k pending webhooks max
```

**What we're capturing:**
- Title + Why (the rationale)
- Status (active/superseded/proposed)
- Which files this decision affects
- Alternatives considered (show we thought it through)
- Implementation notes (how it actually works)
- Constraints (when it breaks)

**Why this works:**
- Markdown + YAML = readable in git, searchable, version-controlled
- Stored in repo next to code (not in a wiki)
- One file per decision = easy to find and update

---

#### 2. File-Level Linking

Decision tracks: "This affects folder X" (not line numbers).

```yaml
affects:
  - src/webhooks/
  - src/queue/
  - tests/webhooks.test.ts
```

**Why file-level, not line-numbers:**
- Line numbers become invalid after refactors
- File-level linking survives code reorganization
- Simple to check: "Did any file in this list change?"

**What counts as "affected":**
- Source files implementing the decision
- Test files verifying the decision
- Config files that depend on the decision

**What doesn't count:**
- Unrelated refactors in the same file (e.g., renaming a helper function)

---

#### 3. Change Detection + Warning System

**Mechanism:** Pre-push git hook (installed via `lattice install-hooks`)

When developer runs `git push`:
1. Hook detects: "Which decisions affect changed files?"
2. For each affected decision: check if already acknowledged (local ack file)
3. If not acknowledged → warn and block push
4. Developer reviews decision, then approves with `lattice decision ack <id>`
5. Ack file created at `.lattice/acks/{email}/{decision_id}` (gitignored)
6. Next push to same files: compare ack file mtime vs decision file commit date. If ack is newer → don't re-warn. If decision changed since ack → warn again.

**Hook installation:**

```bash
lattice install-hooks
# Installs wrapper at .git/hooks/pre-push that execs .lattice/hooks/pre-push
# One-time setup per clone
```

**Example flow:**

```bash
$ git push
⚠️  Decision "async-webhooks" references src/webhooks/ — which you changed.
    Review decision? Still valid? Or needs update?
    Decision file: decisions/async-webhooks.md
    Changed files: src/webhooks/handler.ts, src/queue/worker.ts
    
    Approve as-is: lattice decision ack async-webhooks
    Update decision: edit decisions/async-webhooks.md and commit
    
$ git push
# (same files, same decision, already ack'd) → no warning
# Suppression sticks until decision file or affects: changes
```

**Key properties:**
- Hook installed once per developer (via `lattice install-hooks`)
- Warn once per decision per affected change
- Suppression is per-decision, per-developer (local ack file in `.lattice/acks/`)
- Suppression sticks until decision file changes (no warning fatigue)
- Pre-push = before CI, immediate feedback
- Fallback CLI for teams that disable hooks: `lattice decision check`

## Technical Specification

### Pre-Push Hook Implementation

Location: `.lattice/hooks/pre-push` (tracked in repo)  
Installed to: `.git/hooks/pre-push` (via `lattice install-hooks`)

**Pseudocode:**
```
current_developer = git_config("user.email")
remote_ref = $1 (from pre-push stdin)
local_ref = $2 (from pre-push stdin)

FOR EACH decision in /decisions:
  IF decision.status != "active":
    SKIP (only active decisions warn)
  
  affected_files = YAML.parse(decision.affects)
  changed_files = git_diff_files(${remote_ref}...${local_ref})
  
  # Skip if developer just created this decision in this push
  IF decision.md in changed_files:
    SKIP (developer wrote this decision in this push)
  
  intersection = affected_files ∩ changed_files
  
  IF intersection is empty:
    SKIP (no relevant changes)
  
  # Check if already acknowledged by this developer
  decision_file_commit_date = git_log_date(decision.md, -1)
  ack_file = ".lattice/acks/${current_developer}/${decision_id}"
  ack_file_mtime = file_mtime(ack_file)
  
  IF ack_file_mtime exists AND ack_file_mtime >= decision_file_commit_date:
    SKIP (already acknowledged, no decision changes since ack)
  
  # Not acknowledged or decision changed since ack
  WARN: "Decision '{decision.title}' references changed files"
  PROMPT: "lattice decision ack {decision_id} to suppress"
  # Note: v0.7 warns only, allows push to proceed
  # Blocking (exit code 1) deferred to v1.0 after usage feedback
```

**What this means:**
- Warn on file changes in `affects:` list (only status: active decisions)
- Skip if decision file was just created in this push (developer wrote it)
- Suppression is per-developer (ack file at .lattice/acks/{email}/{id})
- Suppression is per-decision (different devs, different decisions, different acks)
- Re-warn if decision file changes (compare mtime, fresh review needed)

---

### Definition: "Meaningful Change"

Simple rule: **ANY file in `affects:` changed → warn**.

Don't be smart yet. Warn on everything. Later iterations can filter.

```
IF any file in decision.affects changed in this commit:
  WARN (unless ack'd via lattice decision ack)
```

Rationale: False positives are cheap (one command). False negatives are expensive (forgotten constraints). Start permissive.

---

### Suppressing Warnings

Developer reviewed decision, approved it's still valid:

```bash
lattice decision ack async-webhooks
```

This records approval in a local ack file: `.lattice/acks/{user_email}/{decision_id}` (gitignored). Suppression sticks for this developer until the decision file is modified. Then the warning re-triggers (fresh review needed).

**Why local ack file + `lattice decision ack`:**
- Per-decision granularity (surgical, not nuclear)
- Per-developer state (different devs can suppress independently)
- Simple to implement (just file mtime comparison)
- Zero friction (one command)
- Self-healing (re-warns when decision file changes)

## Known Limitations (Accept Before Building)

**1. Adoption depends on discipline, not design**

This only works for teams that:
- Actually write decisions (friction exists, don't lie about it)
- Review warnings (not all will)
- Keep decisions updated (some won't)

If your team is already losing documentation, this won't fix that.

**2. Warning fatigue is real**

Decisions that span large folders = many warnings. You'll see devs ignore warnings. That's expected, not a design bug.

**3. Hook friction for some teams**

Some teams disable hooks entirely. CLI fallback helps, but won't work for everyone.

**4. Real usage will surface edge cases**

Expect issues like:
- Pair programming (shared email confusion)
- CI bots (automation account acks)
- Force-push workflows (acks lost locally)
- Large refactors (warning volume)

These aren't design flaws; they're learning opportunities. Iterate on them during Phase 2.

**5. Adoption is the real test**

The architecture is sound. Success depends on whether your team actually writes decisions and finds value in the warnings. That's behavioral, not technical.

## Implementation Details (Must-Know for Phase 1)

### Ack Storage: Local File (Per-Developer State)

**Decision:** Acks are stored in local `.lattice/acks/` directory (gitignored). They are **NOT shared or pushed**.

**Why:**
- Simpler: No git notes complexity, just file operations
- Matches intent: "per-developer state" (each dev acks independently)
- Correct UX for new developers: Fresh clone has no acks, re-warns on first push to changed files (forces review)

**Implementation:**
- Store as: `.lattice/acks/{user_email}/{decision_id}` (plain file, gitignored)
- File mtime = ack timestamp
- Compare: ack file mtime vs decision file's last git commit date
- If ack_mtime >= decision_commit_date → suppressed
- If decision_commit_date > ack_mtime → warning re-triggers

**Implication:**
- When dev A acks a decision, dev B doesn't see it (each dev reviews independently)
- Acks survive local rebases (file is outside git)
- When dev clones fresh, no acks exist (correct: forces fresh review)
- If dev deletes .lattice/acks, acks are lost (acceptable: rare, acks are re-done on next push)

### Hook Installation: Wrapper Script (Auto-Updating)

**Decision:** `lattice install-hooks` installs a **wrapper script** (not a copy). The wrapper execs the tracked hook.

**Implementation:**

```bash
# .git/hooks/pre-push (installed by `lattice install-hooks`)
#!/bin/sh
exec "$(git rev-parse --show-toplevel)/.lattice/hooks/pre-push" "$@"
```

The tracked hook lives at `.lattice/hooks/pre-push` (git-controlled). Improvements flow through automatically without re-install.

**Install behavior:**
1. Create `.git/hooks/pre-push` with wrapper script that execs `.lattice/hooks/pre-push` (one-time per clone)
2. Make it executable
3. Append `.lattice/acks/` to `.gitignore` if not present
4. Idempotent: running twice detects identical wrapper, exits silently

**Hook collision handling:**
If `.git/hooks/pre-push` already exists (Husky, lint-staged, etc.):
1. `lattice install-hooks` detects existing hook
2. Refuses to overwrite, shows error message
3. Suggests: "Existing hook found. Add this line to the top of your hook: `exec $(git rev-parse --show-toplevel)/.lattice/hooks/pre-push "$@"`"
4. Developer manually chains the hooks

**v1.0+ improvement:** Merge tool to automatically chain multiple hooks (for now, manual).

### Warning Behavior: Warn Only (v0.7)

**Decision:** Unack'd decisions **WARN but allow push** (exit code 0 in v0.7).

**Why:**
- Lightweight guardrail, not enforcement
- Lower friction for early adoption
- Gather usage data before blocking
- Blocking deferred to v1.0 after real-world feedback

**Implication:**
- Push succeeds even with unack'd decisions
- Developer sees warning, can choose to ack or ignore
- CLI fallback (`lattice decision check`) shows same warnings
- v1.0+ will add blocking option if adoption is strong

**Future (v1.0+):**
May add blocking mode with exemptions:
```bash
# v1.0: optional blocking
lattice install-hooks --blocking
# or per-decision
lattice decision set async-webhooks --enforce
```

### Per-Developer Keying: Known Limitation

**Limitation:** Acks are keyed by `git config user.email`. This breaks for:
- Pair programming (both devs commit as one)
- CI bots (commit from a bot account)
- Shared developer accounts

**v0.7 Acceptance:** Document this limitation. v0.8+ can add `--ack-author` flag for explicit keying.

**Workaround (v0.7):** For shared/CI accounts, run `lattice decision ack <id>` before the first push from that account, then CI pushes normally.

### Commit Date vs Author Date

**Rule:** Use commit date (not author date) when comparing against ack file mtime.

**Why:** Commit date is stable across rebases (author date changes). Ensures ack comparisons work correctly through local rebases.

### Ack File Mtime Comparison

**Rule:** Use `>=` for mtime comparison. If ack_mtime >= decision_commit_date, suppression applies.

**Why:** Filesystem mtime precision varies (1s on HFS+, < 1ns on ext4). Using `>=` prevents re-warning if ack and commit happen in same second.

**Alternative:** Store ISO 8601 timestamp inside ack file instead of relying on mtime (more precise, requires parsing).

### Email Path Sanitization

**Rule:** Convert email to safe filesystem path by replacing non-`[a-zA-Z0-9._@-]` characters with `_`.

Example: `user+tag@domain.com` → `user_tag@domain.com` (then stored as `.lattice/acks/user_tag@domain.com/decision_id`)

**Why:** Prevents issues on Windows and filesystems with character restrictions for paths containing encoded or special emails.

### Fallback CLI: lattice decision check

**Behavior:** Manual check for unack'd decisions (when pre-push hook doesn't run).

```bash
lattice decision check
# Outputs: same warnings as pre-push hook
# Exit code: 1 if any unack'd, 0 if clean
```

**Used by:** CI/CD pipelines where pre-push hooks don't run. Developers then run `lattice decision ack <id>` to fix.

## Remaining Implementation Questions (v0.7)

These are implementation-detail questions, not architecture. Resolved during Phase 1 coding:

1. **Hook performance** — Target < 200ms, hard ceiling < 500ms. If hook exceeds 200ms on first measurement, build a decision-to-file cache.

2. **Multi-format decisions** — All decisions must be YAML frontmatter + markdown body (consistent format for parsing).

3. **Cascade warnings** — No cascading in v0.7. If decision A references decision B, only B warns. Cascade logic deferred to v0.8.

4. **Superseded decision handling** — `status: superseded` decisions do NOT warn. Only `status: active` decisions warn. `status: deleted` and `status: proposed` also skip warnings.

## Architecture Overview

```
┌──────────────────────────────┐
│ One-time Setup               │
│ lattice install-hooks        │
│ (installs wrapper script)    │
└──────────┬────────────────────┘
           │
           └─→ .git/hooks/pre-push (wrapper script)
               └─→ execs .lattice/hooks/pre-push (tracked)

┌─────────────────────────────────────────┐
│ Developer Workflow                      │
└─────────────────────────────────────────┘

  Developer Code (.ts/.js)
         │
         ├─→ Edit + Commit
         │
         ├─→ git push
         │
         │ ┌────────────────────────────────┐
         └─→ Pre-push Hook (v0.7)           │
             │                              │

             ├─→ Parse affects: []          │
             ├─→ Check git diff             │
             ├─→ Detect changes             │
             │                              │
             ├─→ Check ack file            │
             │   .lattice/acks/<dev>/<id>  │
             │                              │
             ├─→ If acknowledged & fresh:   │
             │   → ALLOW push               │
             │                              │
             ├─→ If not ack'd or stale:     │
             │   → WARN                     │
             │   → Allow push (v0.7)        │
             │   → Show: Decision file,     │
             │     Changed files            │
             │   → Prompt: lattice decision │
             │     ack <id>                 │
             │                              │
             │ Developer reviews, ack's:    │
             │ lattice decision ack X       │
             │ (records in .lattice/acks/)  │
             │                              │
             └─→ git push (succeeds)        │
             └────────────────────────────┘
         │
         ├─→ Commit in git (ack file in .lattice/acks/)
```

## FAQ

**Q: What if a decision becomes outdated?**
A: Update the decision file, set `status: superseded`, explain why in the file. The pre-push hook will still warn, giving you a chance to decide: keep the warning or remove the decision entirely.

**Q: What if the hook is annoying?**
A: Run `lattice decision ack <id>` to approve the decision. Next push to the same files won't warn (suppression sticks until the decision file changes). If you see the same decision repeatedly and it's always a false positive, update the `affects:` list to exclude the file.

**Q: How is this different from ADRs?**
A: ADRs are a static format (you write it, nobody reads it). v0.7 is a lightweight guardrail: decisions are written, linked to code, and developers get a warning when they're about to change them. It's ADRs + awareness.

**Q: Can I delete a decision?**
A: Yes. Set `status: deleted` in the YAML. The hook won't warn on it anymore. Git history preserves it.

**Q: What about decisions that affect multiple, scattered files?**
A: List them all in `affects:`. If a decision affects 50 files, it's probably too broad—consider splitting into smaller decisions.

**Q: What if multiple decisions warn on one push?**
A: Run `lattice decision ack <id>` for each decision, then push again. Or use `lattice decision ack --all` to acknowledge all warnings in the current batch at once.

**Q: Does my CI bot need to ack decisions?**
A: If `LATTICE_SKIP_HOOKS=1` is set, the hook is skipped and CI can push without acking. Otherwise, the CI account must run `lattice decision ack <id>` once per decision before pushing.
