# Lattice v1.0 — Substrate Spec

**Status:** locked, in active build (started 2026-05-14).
**Predecessor:** v0.8.3 (CLI + findings lifecycle + telemetry).
**Successor of:** [v0.7 fingerprint spec](v0.7-fingerprint-spec.md).

## Motivation

The v0.8.x line proved out the audit + lifecycle methodology: code is truth, findings are tested artifacts. v1.0 takes the same methodology one step further — **architecture state itself is derived from code, not maintained as a separate document.**

Why now: in autonomous Claude-driven development, doc rot is not a slow leak, it is an immediate confidence-amplifier for wrong information. Stale `AGENTS.md`, `ADR.md`, or `README.md` files actively make agents perform worse (ETH 2026: bloated context reduces success ~3%, +20% cost). The substrate dissolves the problem by removing the maintained doc entirely; every consumed view is regenerated from code on read.

## Single user, single orchestrator (2026 scope)

v1.0 is **opinionated for one user (Isaam) and one orchestrator (Claude Code)**. No multi-tenant, no public release for one year, no multi-orchestrator abstraction. This scope unlocks aggressive design choices that would be wrong for a public product.

Re-evaluate scope at v2.0.

## Core principles

1. **Code is the only authoritative artifact.** Everything else is a derived view.
2. **No prose docs outside `.lattice/`.** External READMEs and Notion pages are advisory only, never consulted by Claude during sessions.
3. **Mechanical truth cannot drift.** Invariants, context, traces, drift scores — regenerated on every read.
4. **Narrative truth drifts loudly.** ADRs (the only hand-written piece) emit Lattice findings when stale; never silent.
5. **Backward compatible.** v0.8.x projects (jiive) keep working unchanged via `mode: classic`.

## Operating modes

`.lattice/config.yml` adds a `mode:` field:

| Mode | Behavior | Initial fit |
|---|---|---|
| `classic` | v0.8.x behavior. Findings + CLAUDE.md sync. No MCP, no invariants, no traces. | jiive backend |
| `substrate` | Full v1.0: invariants + drift bounds + traces + ADRs + MCP. | riseCraft rebuild + 2 recovery projects |
| `hybrid` | Findings + ADRs + MCP, no invariant derivation. Lower compute, no drift score. | Mid-firming-up projects |

Default for new projects: `substrate`.
Existing v0.8.x projects without `mode:` set: treat as `classic`.

## ADR schema (`.lattice/decisions/<n>-<slug>.yml`)

```yaml
# Required
id: 0001-supabase-migration
slug: supabase-migration
title: "Migrate backend from Node/Express on AWS to Supabase"
status: in_progress | active | superseded | reversed
created_at: 2026-05-14T12:00:00Z
created_by: claude-code/session-<id>
because: "$3k/month AWS spend with $0 revenue; standard CRUD+auth fits Supabase"

# Optional
still_valid_as_of: 2026-05-14       # required for active ADRs older than 30 days
superseded_by: 0042-some-newer-slug # set when superseded
reverses: 0017-some-prior-slug      # set when reversing a prior decision

# Citations — Lattice audits these (file:line ranges)
cited_files:
  - path: supabase/migrations/20260514000001_schema.sql
    lines: 1-200
    purpose: "the new schema implementing the decision"
  - path: lib/core/services/api_client.dart
    lines: 1-50
    purpose: "the frontend boundary affected by this decision"

# Relations
relates_to:
  - finding: HIGH-payments-flow-payment-state-never-saved
  - finding: HIGH-auth-dual-401-interceptor-race

# Discussion (free-form, append-only)
discussion: |
  Considered Hasura + custom Node. Rejected: still requires backend infrastructure.
  Considered keeping Node minimal. Rejected: still incurs $200+/mo for the smallest
  meaningful deployment + monitoring overhead.
```

## Invariants schema (`.lattice/invariants/HEAD.yml`)

Auto-derived from code via tree-sitter. Regenerated on every commit or `lattice invariants regen`.

```yaml
# Hash-pinned to the commit
commit: 9e55a0e
derived_at: 2026-05-14T12:00:00Z
derived_by: lattice/invariants-engine@1.0

# Structural facts
modules:
  - path: lib/features/auth
    public_exports: [AuthProvider, signIn, signOut]
    depends_on: [lib/core/services/api_client.dart]
  - path: supabase/functions/razorpay-create-order
    runtime: deno
    secrets_used: [RAZORPAY_KEY_ID, RAZORPAY_KEY_SECRET]

# Contracts (interfaces / API surfaces)
contracts:
  - kind: edge_function
    name: razorpay-create-order
    auth: required
    inputs: { amount: number, currency: string }
    outputs: { order_id: string, key: string }
  - kind: db_table
    name: user_subscriptions
    rls_enabled: true
    columns: [...]

# Invariants (machine-checkable rules derived from code)
invariants:
  - id: inv-auth-required-admin
    rule: "Every Edge Function under /admin/* MUST verify auth.uid()"
    derivation: "scanned supabase/functions/admin-*/index.ts for auth check pattern"
    holds: true
    cited_files: [supabase/functions/admin-ops/index.ts:8-30]
  - id: inv-no-frontend-secrets
    rule: "lib/ MUST NOT contain any reference to *_KEY_SECRET or service-role keys"
    derivation: "grep across lib/ for known secret patterns"
    holds: true
```

```yaml
  - step: 1
    type: plan_drafted
    summary: "Edge Function for resize + Flutter UI + migration + RLS policy"

    proposal_hash: abc123def

    verdict: pass
  - step: 3
    type: file_write
    path: supabase/functions/upload-avatar/index.ts

  - step: 4
    type: verify_action
    verdict: pass
  - step: 5
    type: record_adr

    type: close_finding
    slug: MEDIUM-user-profiles-no-avatar-support

```

## MCP endpoint surface

The Lattice MCP server (`lattice mcp serve`) exposes these calls to Claude Code:

| Endpoint | Returns | Use |
|---|---|---|
| `get_context()` | active invariants summary + open findings + active ADRs + drift status | session start |
| `get_invariants(file?: path)` | invariants relevant to a file or whole repo | before editing |
| `verify_action(diff_text)` | verdict + violated invariants list + suggested fixes | before applying patch |
| `record_trace_step(session_id, step)` | ack | every meaningful action |
| `record_adr(slug, body)` | adr_id | when making architectural decisions |
| `close_finding(slug, reason, commit?)` | ack | as fixes ship |
| `list_findings(filter?)` | filtered findings | triage |
| `replay_session(session_id)` | trace + verification | postmortem |
| `drift_score(session_id?)` | numeric score + breakdown | per-session quality gate |

All endpoints idempotent + return structured YAML. No prose responses to model — pure data the orchestrator parses.

## CLI surface (delta from v0.8.3)

New commands (~5 added):

```
lattice mode <classic|substrate|hybrid>   # switch mode for this project
lattice mcp serve                          # run MCP server (foreground)
lattice context                            # what Claude sees at session start (debug)
lattice invariants regen                   # force re-derive (auto-runs in CI)
lattice invariants check                   # diff current invariants vs HEAD baseline
lattice decide <slug>                      # create or amend ADR
lattice decisions list [--status active]   # list ADRs
lattice trace <session-id>                 # inspect or replay a trace
lattice drift                              # show drift score for recent activity
```

Existing v0.8.x commands unchanged.

## Compatibility guarantees

1. **`.lattice/findings/*.yml` schema is frozen.** ID format, lifecycle fields, close reasons — all unchanged. Existing jiive + riseCraft findings keep working byte-for-byte.
2. **`lattice list/show/close/sync/stats/doctor/verify --rerun-grep` behavior frozen.**
3. **CLAUDE.md regen marker block format frozen.**
4. **`mode:` defaults to `classic` for any project without the field.** Zero-friction for jiive.
5. **`lattice update --self` from v0.8.x to v1.0 has zero data migration.** New directories created lazily on first use of substrate features.

## Build order (work breakdown)

1. **Mode switch + ADR foundation** — `mode:` field, `decisions/` dir, ADR schema, `lattice decide` + `lattice decisions list`. Ships as v0.9.0.
2. **Tree-sitter invariant derivation** — Dart + TypeScript + SQL parsers, invariant emission. Ships as v0.9.1 (substrate-mode opt-in).
3. **MAT trace recording + replay** — `lattice trace` + `lattice replay`. v0.9.2.
4. **MCP server** — Node.js, exposes the 9 endpoints. v0.9.3.
5. **Drift scoring + bounds enforcement** — drift score per session, bound enforced by `verify_action`. v1.0-rc1.
6. **Dogfood on riseCraft rebuild + jiive optional migration.** Ship v1.0 stable.

Telemetry stays as v0.8.0 design throughout. `LATTICE_OWNER_MODE=1` env added so personal projects flip telemetry default ON without per-project config.

## Open questions

1. **How aggressive is `verify_action` blocking?** Hard fail vs warning depends on invariant criticality. Spec the criticality taxonomy in v0.9.0 implementation.
2. **Should `lattice context` cache?** Today's answer: no, regenerate every call. Revisit if MCP call latency becomes a friction in long Claude sessions.
3. **Drift score formula.** Start with ABC's Jensen-Shannon Divergence baseline (`D* < 0.27` from research); tune to riseCraft real-world usage.
4. **Trace storage limit.** Per-project trace dir could grow large. Auto-prune traces older than N days? Compress to bundle file? Decide before drift scoring needs trace history.

## Why this approach over AGENTS.md / ADR-only

Surveyed in 2026 research (May 14 session):
- AGENTS.md is becoming the public-product standard, but ETH 2026 study shows context files often *reduce* success.
- Knowledge graphs (Cognee, lat.md, Code Graph Models) — useful, but still treat docs as a separate artifact that can drift.
- ABC (Agent Behavioral Contracts) — formal drift bounds, but no project-side substrate to bound against.

Lattice v1.0 combines (1) the *derive-from-code* discipline that prevents drift, (2) the ABC drift-bounds mathematics that detects what slips through, and (3) the MCP standard interface that lets Claude consume both. Closest existing tool: none. The combination is novel for the single-user single-orchestrator case.

## Locked decisions

- Tree-sitter for parsing (Dart + TS + SQL minimum)
- Node.js for MCP server runtime
- YAML for all schemas (consistent with existing findings)
- ABC drift-bounds math as starting point (tunable threshold)
- Single-user single-orchestrator scope until 2027 review
- No external doc authority beyond `.lattice/`
- ADR `still_valid_as_of` field is mandatory after 30 days; missing = Lattice finding
